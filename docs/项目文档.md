# 基于多模态 Agent 的学术视频+论文助手

## 系统概述

**关键词：** Multi-Agent、Graph RAG、LangGraph、MCP、长期记忆、RAGAS、Prompt Engineering 等

---

## 定义

1. **学术视频**：有关前沿论文的讲解视频，包括会议学术报告、Oral Presentation 等。视频讲解形式常为 PPT。

---

## 功能概览

用户提供视频链接（B站或 Youtube），系统搜索、下载并处理视频与原文献；而后用户提出有关视频内容或原文献内容的问题，系统负责带定位原文的解答。系统具备跨会话长期记忆，可记录用户偏好和历史问答要点。

## 模块设计与实现

### 对话 Agent

用户交互的入口，完整节点流程：

```
load_memory → normalize_input → brain_agent → render_response → save_memory → log_eval_sample
```

**节点职责：**

- `load_memory`：从长期记忆 Store 中检索与当前 query 相关的历史记忆，注入 `long_term_memory` 字段
- `normalize_input`：将用户原始输入规范化为自包含的 `normalized_query`，解析指代、补全上下文，同时结合长期记忆理解用户偏好
- `brain_agent`：子图，负责检索与推理（见大脑 Agent 章节）
- `render_response`：将检索上下文改写为流畅自然语言，附带来源定位（如"论文第 3 页"、"视频 slide 7 约 2:30"）
- `save_memory`：调用 LLM 从本轮对话提取 0-3 条记忆写入 Store
- `log_eval_sample`：将本轮 `(query, contexts, answer, reranker_scores)` 写入评估日志数据库，供离线 RAGAS 评测使用

---

### 大脑 Agent

系统的推理与规划核心，采用 **Plan-and-Execute + Reflection** 架构，包含规划者（Planner）和执行者（Executor）两个 LLM 实例。

#### 执行流程

1. 规划者收到 `normalized_query` 后，结合当前已加载资源和可用工具列表，生成结构化的分步执行计划（JSON 格式，含 `step_id`、`tool_name`、`tool_args`）
2. 对于依赖前步结果的参数，Planner 使用占位符（如 `<formula from step 1>`），由 Executor 在执行时替换
3. Executor 逐步执行计划：每步执行前，LLM 结合 scratchpad（前步结果日志）动态精炼 `tool_args`，解析占位符并融入实际数据
4. 工具函数执行，结果写入 scratchpad
5. Executor 评估执行结果：成功则推进到下一步；失败则根据重试预算决定重试同一步骤，或将错误报告升级给规划者
6. 规划者收到错误报告后，结合原计划和错误信息重新规划（最多 3 次）

#### 状态管理

所有节点共享单一 `OverallState`（方案 A），关键字段包括：

- `messages`：多轮对话历史（`add_messages` reducer，追加不覆盖）
- `plan` / `current_step_index` / `replan_count`：规划执行进度
- `executor_scratchpad`：每步执行结果日志，供 Executor LLM 读取
- `retrieved_context`：Brain Agent 汇总的检索结果，交由 Chat Agent 渲染
- `long_term_memory`：当前轮注入的历史记忆字符串，供 `normalize_input` 使用
- `status_message`：当前执行阶段提示，通过 SSE 推送至前端

#### 图结构

LangGraph 子图组合：Chat Agent 子图内嵌 Brain Agent 子图，Brain Agent 节点包括：

- `plan → execute_step → evaluate_step`，循环至 `assemble_context`
- 路由函数 `route_after_plan` / `route_after_evaluate` 控制重试、重规划、放弃三条路径
- 每轮对话开始时由根图的 `reset_brain_state` 节点清空上轮残留状态

### GraphRAG

处理视频及对应的论文 PDF，存入数据库中，并构建图。在用户提出有关内容的 query 后，进行 RAG。

#### 数据处理

**1. 视频**
- 提取出带时间戳的讲解字幕（Whisper-large-v3-turbo）
- 提取出画面关键帧。由于视频大多是 PPT 讲解，因此以一页 PPT 为一个关键帧
- 一个 PPT 页面及其对应的讲解字幕为一组数据（VideoSegment）
- PPT 画面通过 Gemini Flash 提取文字和画面描述，存为文本形式
- 视频字幕若为中文，直接存入节点原文字段；使用多语言嵌入模型处理中英混合内容

**2. 论文 PDF**
- 通过 Docling 将原文分成 Chunks，每个 Chunk 为一个 PaperChunk 节点

#### 数据库设计

**Milvus Lite（向量数据库）**

使用 BGE-M3 嵌入模型，同时生成 Dense 向量和 Sparse 向量，支持混合检索。

PaperChunk Collection 字段：
- `chunk_id`（主键）、`page`、`section`、`source`
- `content`：原文内容，用于检索
- `dense_vector`：BGE-M3 Dense 输出，dim=1024
- `sparse_vector`：BGE-M3 Sparse 输出，用于词汇匹配

**Neo4j（图数据库）**

节点类别：
- `PaperChunk`：论文原文块，一个 Chunk 一个节点
- `VideoSegment`：视频 PPT 页面节点，包含字幕和 PPT 内容
- `Entity`：从视频和论文内容中提取的学术实体，分为 Concept、Method、Metric、Equation、Experiment 五类

节点间关系：
- `NEXT`：PaperChunk 之间、VideoSegment 之间的顺序关系
- `MENTIONED_IN`：Entity 与 PaperChunk/VideoSegment 之间的提及关系，带有 `importance` 和 `mention_context` 属性
- `CORRESPONDS_TO`：PPT 页面与论文块之间，若共同实体数量 ≥ 4 则建立对应关系

#### 实体提取与图构建

**实体提取**：使用 Gemini Flash 对每个 PaperChunk 和 VideoSegment 提取学术实体，全局维护已知实体列表保证名称一致性，支持断点续跑。

**实体筛选**：按组合条件过滤低价值实体：
- 保留：跨源实体（论文+视频均出现）且 `avg_importance ≥ 0.5`
- 保留：高频实体（`mentions ≥ 5`）且 `avg_importance ≥ 0.5`
- 保留：`max_importance ≥ 0.9` 且 `mentions ≥ 3`
- 过滤：单次提及的公式变量符号（Equation 类型且 `mentions ≤ 2`）

**对应关系建立**：统计每对 (PaperChunk, VideoSegment) 的共同实体数量，阈值 ≥ 4 建立 `CORRESPONDS_TO` 边。

#### RAG 流程

系统提供两种检索模式，由 Brain Agent 的 Planner 根据 query 类型自动选择：

**模式一：向量语义检索（`rag_retrieve`）**

适用于开放式语义问题，如"GRPO 的核心贡献是什么"。

1. 在 Milvus 中同时进行 Dense（语义）和 Sparse（词汇匹配）混合检索，RRF 融合排序，召回 Top-K 节点
2. 图扩散（可选）：沿 `NEXT` 关系获取相邻 Chunk，解决 Chunk 边界切割导致内容不完整的问题。图扩散时同步记录 `neighbour_to_primary` 映射，用于 rerank 后精确过滤邻居节点
3. 跨源标注：查找召回节点之间的 `CORRESPONDS_TO` 关系，在 `summary` 字段注明对应的视频页/论文块
4. **Reranker**：使用 BGE-Reranker-v2-M3 对 primary chunks 重新打分排序，截断至 top-k；邻居节点跟随其 primary chunk 是否存活决定去留

**模式二：结构化图查询（`graph_rag_retrieve`）**

适用于涉及明确结构关系的查询，直接走 Cypher 精确查找，不经过 Milvus。提供六种模板：

| 模板 | 适用场景 | 参数示例 |
|---|---|---|
| `lookup` | 按属性精确定位节点（视频页码/时间戳、论文页码/章节、实体名） | `{"node_type": "video", "by": "page", "value": 5}` |
| `video_to_paper` | 给定视频页/时间戳，通过 CORRESPONDS_TO 找对应论文块 | `{"page": 10}` 或 `{"time": 300}` |
| `paper_to_video` | 给定论文页/章节，找对应视频片段 | `{"page": 3}` 或 `{"section": "3.2"}` |
| `entity_mentions` | 给定实体名，找所有提及该实体的论文块和视频片段（按 importance 排序） | `{"entity_name": "GRPO"}` |
| `entity_co_occur` | 给定两个实体名，找两者共同出现的节点 | `{"entity1": "PPO", "entity2": "GRPO"}` |
| `sequential` | 从指定论文块出发，沿 NEXT 链取后续 N 块 | `{"page": 2, "hops": 3}` |

#### 跨步依赖处理

对于需要前步结果才能确定参数的多步查询（如"视频第 8 页提到的公式在论文里有推导吗"）：

- Planner 在规划时对依赖前步的参数使用占位符，如 `<formula from step 1>`
- Executor LLM 在执行每步前读取 scratchpad，将占位符替换为前步的实际结果，再调用工具
- 这使得系统可以完成真正的跨模态、跨步骤推理

### 长期记忆机制

系统支持跨会话的长期记忆，记录用户偏好、关注方向和历史问答要点，在后续对话中自动注入上下文。

#### 存储架构

采用双层设计，两者通过 `PersistentSqliteStore` 统一封装：

```
写入路径（save_memory）：
  store.aput() → abatch([PutOp])
    ├─ SQLite：INSERT INTO long_term_memories   （持久化，重启不丢失）
    └─ InMemoryStore：建向量索引               （支持语义搜索）

读取路径（load_memory）：
  store.asearch(query) → abatch([SearchOp])
    └─ InMemoryStore：BGE-M3 向量相似度检索    （返回 top-k 相关记忆）

重启恢复（setup）：
  SQLite → 逐条 aput → InMemoryStore          （重建向量索引）
```

**设计要点**：所有 LangGraph store 操作经由 `abatch()` 统一路由，`PersistentSqliteStore` 只需 override 此一个方法即可同时保证持久化和向量搜索能力。

#### 记忆内容

每条记忆为一个短文本，分两类：

- **preference**：用户偏好，如"用户更关注数学推导细节，不喜欢泛泛概述"
- **fact**：用户提及的关键事实，如"用户在本次对话中重点关注 GRPO 与 PPO 的 reward shaping 差异"

#### 存储规格

- 命名空间：`("user", "default_user", "memories")`（单用户系统）
- 嵌入模型：BGE-M3 dense 向量（dim=1024，复用 RAG 已初始化的单例）
- 持久化路径：`MEMORY_DB_PATH` 环境变量，默认 `./data/memories.db`
- 每轮提取条数：0-3 条（由 LLM 判断是否有值得记录的内容）
- 检索条数：top-5 相关记忆

#### 节点流程

```
load_memory
  ↓  asearch(raw_user_input, limit=5)
  ↓  注入 state.long_term_memory

normalize_input
  ↓  prompt 包含记忆上下文，LLM 结合用户偏好理解问题

... （brain_agent + render_response）...

save_memory
  ↓  LLM 提取本轮要点
  ↓  aput 写入 Store（SQLite + InMemoryStore 同步）
```

### MCP 集成

系统通过 MCP（Model Context Protocol）接入外部工具服务。

#### 架构设计

采用**标准 MCP 用法**：在 FastAPI 启动时通过 `MultiServerMCPClient` 建立持久连接，将 MCP server 暴露的所有工具动态注入 Brain Agent 的工具注册表（`ALL_TOOLS` / `TOOL_MAP`），由 Planner LLM 自主选择调用哪个工具。

```
FastAPI lifespan 启动
  → init_mcp()
    → MultiServerMCPClient 启动 MCP server 子进程（npx）
    → 加载所有 MCP tools
    → 注入 ALL_TOOLS / TOOL_MAP

Planner 生成计划
  → tool_name: "search_repositories"（LLM 自主选择）

execute_step
  → TOOL_MAP["search_repositories"].ainvoke(args)
    → 通过已有持久连接调用 MCP server
```

与旧方案（每次调用冷启动 npx 进程、硬编码调用特定函数）相比：
- **连接复用**：进程常驻，无冷启动延迟
- **LLM 自主决策**：Planner 可见全部 MCP 工具，而非被包装成单一接口
- **易扩展**：新增 MCP server 只需在配置中加一条记录

#### 当前接入的 MCP Server

| Server | 工具来源 | 主要用途 |
|---|---|---|
| `@modelcontextprotocol/server-github` | GitHub 官方 | 搜索代码仓库、读取文件、查看 commits 等 |

#### 生命周期管理

```python
# server.py lifespan
await init_mcp()   # 启动：建立连接，加载工具
yield
await close_mcp()  # 关闭：终止 MCP server 子进程
```

### RAG 评估

系统内置轻量评估机制，支持运行时数据采集和离线 RAGAS 批量评测。

#### 运行时日志采集

`log_eval_sample` 节点在每轮对话末尾将以下字段写入 SQLite（`EVAL_LOG_PATH`，默认 `./data/eval_log.db`）：

| 字段 | 说明 |
|---|---|
| `query` | 规范化后的用户问题 |
| `contexts` | 检索到的 chunk 文本列表 |
| `answer` | 最终生成的回答 |
| `reranker_scores` | 每个 chunk 的 reranker 分数 |
| `resource_types` | 每个 chunk 的来源类型（paper_chunk / video_slide 等） |

日志写入为非阻塞操作，异常静默处理，不影响主流程。

#### 离线评测（evaluate.py）

```bash
python evaluate.py              # 评测全部样本
python evaluate.py --limit 50   # 评测最近 50 条
python evaluate.py --no-ragas   # 仅看检索统计，不调用 LLM
python evaluate.py --output results.json
```

#### 评测指标

全部为 **reference-free**（无需人工标注标签）：

| 指标 | 含义 | 判断方式 |
|---|---|---|
| **Faithfulness** | 回答中每个声明是否有检索内容支撑 | LLM 逐句核查 |
| **Answer Relevancy** | 回答是否真正回答了问题 | LLM 打分 + BGE-M3 嵌入相似度 |
| **Context Relevancy** | 检索到的 chunks 有多少比例与问题相关 | LLM 打分 |
| **Reranker score 分布** | top-1 均值、全体均值/标准差、来源类型分布 | 直接统计，免费 |

**评测 LLM**：deepseek-chat（复用项目已有配置）
**嵌入模型**：BGE-M3（复用 RAG 已初始化的单例）

#### 指标解读参考

- Faithfulness < 0.7：`render_response` 存在幻觉风险，考虑加强 prompt 约束
- Answer Relevancy < 0.7：回答偏题，考虑优化 `CHAT_RENDER_PROMPT`
- Context Relevancy < 0.5：检索引入了较多噪音，考虑降低 `top_k` 或提高 reranker 截断阈值
- Reranker top-1 均值 < 0.3：检索质量整体偏低，考虑调整 embedding 策略或 query 改写
